# Spring Batch Schedulers

A Spring Boot mini-project demonstrating how to trigger **Spring Batch jobs periodically** using `@EnableScheduling` and `@Scheduled` annotations, backed by a PostgreSQL database.

## What This Project Demonstrates

- Combining **Spring Batch** with **Spring Scheduling** (`@Scheduled`) to run batch jobs on configurable cron schedules
- Defining multiple independent batch jobs (three in this project) in a single application
- Using **chunk-oriented processing** with `JpaCursorItemReader`, custom `ItemProcessor`s, and `JpaItemWriter`
- Using a **Tasklet step** for non-chunk work (pre-computing report aggregates)
- Persisting an **audit log** for every job execution (scheduled and manual)
- Exposing a **REST API** to query job history, products, reports, and trigger jobs manually
- **Flyway** for schema lifecycle management (all tables, including Spring Batch meta-data tables)
- **Unit tests** for domain logic with JUnit 5 (no Spring context needed)
- **Integration tests** with Testcontainers (real PostgreSQL in Docker)

## Architecture

### Three Batch Jobs

| Job | Schedule (default) | What it does |
|---|---|---|
| `priceRefreshJob` | Every 30 seconds | Applies a configurable price adjustment factor to every product |
| `inventoryAuditJob` | Every 60 seconds | Sets/clears the `lowStock` flag when `stockQuantity < 10` |
| `reportGenerationJob` | Every 2 minutes | Generates per-category summary reports (count, stock, avg price) |

### Domain Model

```
products           – product catalog (name, category, price, stockQuantity, lowStock)
job_audit_logs     – one row per job execution (jobName, status, triggerType, timing)
product_reports    – per-category snapshot rows generated by reportGenerationJob
BATCH_*            – Spring Batch meta-data tables (managed by Flyway V4 migration)
```

## Requirements

- **Java 21+**

## Running with Docker Compose

Docker Compose is the recommended way to run the full stack.

### Start the application

```bash
docker compose up --build
```

This starts:
- `postgres` — PostgreSQL 16 on port `5432`
- `app` — Spring Boot application on port `8080`

Flyway automatically applies all migrations (V1–V4) on first startup and seeds 20 sample products.

### Stop the application

```bash
docker compose down
```

### Stop and remove all data (fresh start)

```bash
docker compose down -v
```

### Inspect the database directly

```bash
psql -h localhost -p 5432 -U batchschedulers -d batchschedulersdb
```

## Scheduler Configuration

The cron schedules are configurable via application properties or environment variables:

| Property | Env variable | Default | Description |
|---|---|---|---|
| `batch.schedule.price-refresh-cron` | `BATCH_SCHEDULE_PRICE_REFRESH_CRON` | `0/30 * * * * *` | Every 30 seconds |
| `batch.schedule.inventory-audit-cron` | `BATCH_SCHEDULE_INVENTORY_AUDIT_CRON` | `0 * * * * *` | Every 60 seconds |
| `batch.schedule.report-generation-cron` | `BATCH_SCHEDULE_REPORT_GENERATION_CRON` | `0 0/2 * * * *` | Every 2 minutes |
| `batch.price-refresh.adjustment-factor` | `BATCH_PRICE_REFRESH_ADJUSTMENT_FACTOR` | `1.02` | 2% price increase |

### Example: change schedule via Docker Compose override

Uncomment and edit the environment variables in `docker-compose.yml`:

```yaml
environment:
  BATCH_SCHEDULE_PRICE_REFRESH_CRON: "0 0/5 * * * *"   # every 5 minutes
  BATCH_PRICE_REFRESH_ADJUSTMENT_FACTOR: "1.00"          # no price change
```

## How Scheduling Works

```
@SpringBootApplication
@EnableScheduling              ← activates Spring's scheduling infrastructure
class SpringBatchSchedulersApplication { ... }

@Component
class BatchJobScheduler {
    @Scheduled(cron = "${batch.schedule.price-refresh-cron:0/30 * * * * *}")
    void schedulePriceRefreshJob() {
        runJob(priceRefreshJob, "SCHEDULED");   ← launches the Spring Batch job
    }
}
```

Each `@Scheduled` method:
1. Writes a `JobAuditLog` record with status `RUNNING`
2. Builds `JobParameters` with a unique `run.id` (timestamp) so Spring Batch allows repeated runs
3. Calls `JobLauncher.run(job, params)` — **synchronous** by default, blocks until the job completes
4. Updates the audit log with the final status and finish time

## Database Schema

```
V1 – products table
V2 – job_audit_logs table
V3 – product_reports table
V4 – Spring Batch meta-data tables (BATCH_JOB_INSTANCE, BATCH_JOB_EXECUTION, etc.)
```

Flyway scripts are in `src/main/resources/db/migration/`.
